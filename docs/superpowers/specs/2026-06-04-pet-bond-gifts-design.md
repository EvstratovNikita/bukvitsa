# Привязанность Букли — коллекция подарков за кормление

**Дата:** 2026-06-04
**Статус:** утверждён, готов к плану реализации

## Контекст и проблема

Сейчас единственный выхлоп кормления питомца — ускорение регена энергии
(сытость → скорость, ×1…×2). Энергия в игре дешёвая и обильная (20 монет,
реклама, доп-режимы), поэтому кормление слабо мотивировано. Нужна
**неденежная** награда/механика, которая реально стимулирует регулярно
кормить Буклю.

**Решение:** добавить петлю «Привязанность» — пока Букля сыта, со временем
растёт привязанность; на максимуме она приносит **эксклюзивный косметический
подарок** (фон/стиль клеток, которые нельзя купить). Подарки бесплатны —
«платой» служит вложение игрока в кормление. Реген энергии от сытости
**остаётся** как отдельный мгновенный бонус.

Цель — ежедневный возврат и забота о питомце + коллекционная мотивация,
рассчитанная на долгую жизнь игры (лайв-опс: новые подарки раз в 1–2 месяца).

## Принятые решения

- Модель петли: **уход во времени** (интеграл сытости), не спам-кормление.
- Награда: **коллекция эксклюзивной косметики** (не монеты, не сила).
- Старт: **4 предмета**; каталог расширяемый (дописывание элементов = лайв-опс).
- Подарки **бесплатные**.
- Косметика подарка **не применяется автоматически** — игрок применяет вручную.
- Реген энергии от сытости сохраняется без изменений.

## Архитектура

### Хранение данных (без миграций БД)

Bond и собранные подарки — **клиент-авторитетные**, живут в `prefs`
(свободный jsonb, который клиент пишет сам; недавно `prefs` сделан
client-authoritative при reconcile, web `prefs`-колонка + облако Яндекса):

- `prefs.petBond: number` — прогресс к следующему подарку (0…`BOND_PER_GIFT`).
- `prefs.petBondTickAt: string` — ISO последнего пересчёта bond.
- `prefs.petGifts: string[]` — id уже полученных подарков (по порядку выдачи).

Почему не в `pet` jsonb: `pet` — серверный (пишется RPC `award_win`/`buy_treat`),
reconcile его перетирает. У подарков нулевая экономическая ценность →
серверная авторитетность и анти-чит не нужны. `prefs` синкается уже сейчас.

**Важно — долговечность (БД + смена устройства).** `prefs` хранится не локально,
а в колонке `user_stats.prefs` (Supabase) и в облаке Яндекса (`player.setData`),
поэтому переживает очистку localStorage и смену устройства. НО недавний фикс
«слёта темы» сделал `prefs` целиком **client-authoritative** при reconcile
(локальное побеждает серверное). Для собранных подарков это опасно: на новом
устройстве/после очистки localStorage локальный `petGifts` пуст и при «локальное
побеждает» **затрёт серверный список** → потеря коллекции.

Поэтому reconcile для `prefs` меняем с «локальное побеждает» на **умное слияние
по ключам**:
- `petGifts` → **объединение (union)** локального и серверного списков — никогда
  ничего не теряет, чинит и гонку позднего `userId`, и свежее устройство.
- `petBond` → **max** локального и серверного.
- `theme`, `enterOnLeft` → остаются local-authoritative (фикс слёта темы цел).
- `activeBackground`/`activeCellStyle` → остаются local-authoritative (как сейчас);
  на новом устройстве применённый подарок не переедет, но сам он будет в
  коллекции (через union) и его можно применить заново — приемлемо.

Затронуть оба пути синка:
- `src/hooks/useRemoteSync.js` — в reconcile заменить блок, где сейчас
  `prefs: s.prefs` (целиком локальное), на по-ключевое слияние выше.
- `src/hooks/useYandexSync.js` — при загрузке облака так же union/max для
  `petGifts`/`petBond` (а не слепое `{...s, ...data}`).

### Накопление привязанности

Считается на клиенте, time-based, идемпотентно. **Темп фиксированный — +1 очко
за каждые 5 минут, пока Букля сыта (голод > 0)**, и НЕ зависит от того, насколько
полна шкала голода (минимальный корм тоже копит, просто быстро кончается).

Ключевая величина — **время сытости** в прошедшем окне (сколько минут голод был
> 0), выводится из линейного распада голода (`HUNGER_DECAY_PER_HOUR = 30`/час):

```
decayPerMin   = HUNGER_DECAY_PER_HOUR / 60                 // 0.5 ед/мин
elapsedMin    = (now - petBondTickAt) / 60000
minutesUntilEmpty = hungerStart > 0 ? hungerStart / decayPerMin : 0
fedMinutes    = min(elapsedMin, minutesUntilEmpty)         // время, пока голод > 0
bond         += fedMinutes / BOND_MINUTES_PER_POINT        // +1 за каждые 5 мин
bond          = min(bond, BOND_PER_GIFT)                   // не стокпайлим
petBondTickAt = now
```

`hungerStart` — сытость на момент `petBondTickAt` (т.е. до распада за окно). Bond
держим числом (накапливаем дробно `fedMinutes/5`), для UI показываем процент.

Константы (тюнятся; `BOND_PER_GIFT` — главный рычаг сложности):
- `BOND_MINUTES_PER_POINT = 5` → 12 очков/час сытости.
- `BOND_PER_GIFT = 500` → ~41ч сытости ≈ **5–6 дней** регулярного ухода/подарок.
  (Ориентиры: один полный корм ≈ ~40 очков; ~8ч сытости/день ≈ ~96 очков/день.)

Правила:
- Темп не зависит от уровня голода: при сытости 5 и 100 идёт одинаковые +1/5мин,
  разница лишь в том, КАК ДОЛГО голод останется > 0 (больше корм → дольше сыта →
  больше очков). Минимальный корм даёт немного очков и затухает.
- Голодная (голод = 0) Букля прогресс не копит; offline голод быстро обнуляется
  (~3.3ч с полного до нуля) → AFK/спам неэффективны.
- Если **все доступные подарки уже собраны** (`petGifts` покрывает весь каталог)
  — bond **заморожен** (не растёт), UI показывает «Скоро новые подарки».
- Пересчёт встроить рядом с reconcile голода/энергии в `useStats`
  (читает уже отреконсиленную сытость; пишет `prefs.petBond`/`petBondTickAt`
  через существующий `setPref`/патч prefs).

### Каталог подарков

Новый файл `src/data/petGifts.js` — упорядоченный расширяемый список.
Форма элемента:

```js
{ id, type: 'background' | 'cells', icon, name, desc, payload }
```

- `type:'background'` → `payload.gradient` (как у фонов в `shopItems.js`).
- `type:'cells'` → стиль применяется body-классом `cell-style-<id>` + CSS-правила
  (как `cells-neon`/`cells-emerald`).

Стартовые 4 (концепты; финальные градиенты/CSS — на реализации):
1. `gift-night` 🌙 Фон «Совиная ночь» — глубокая ночь, луна, блики глаз.
2. `gift-feather` 🪶 Фон «Перо Букли» — мягкий перьевой узор.
3. `gift-moonsilver` ✨ Стиль клеток «Лунное серебро» — серебристый диагональный
   блик на верных клетках (по образцу `shine-diagonal`).
4. `gift-owleye` 🦉 Стиль клеток «Совиный глаз» — тёплое янтарное свечение верных.

### Применение косметики подарков

Расширить резолвер косметики, чтобы id подарков работали наравне с магазинными:
- `src/hooks/useShopTheme.js`: фон сейчас берётся `getItem(activeBackground)`
  только из `shopItems`. Ввести общий резолвер (shopItems ∪ petGifts), напр.
  `resolveCosmetic(id)`; фон-подарок отдаёт `payload.gradient`.
- Для стилей клеток: список классов в `useShopTheme` (`all = [...]`) дополнить
  id стилей-подарков (или собирать динамически из обоих каталогов) + добавить
  CSS-правила для них в `index.css`.
- Применение через существующие `activeBackground`/`activeCellStyle` (уже
  client-authoritative). Владение подарками определяется `prefs.petGifts`
  (а не `inventory`, который серверный) — поэтому подарочная косметика
  не зависит от серверного инвентаря.

### UI и петля выдачи

Секция «Подарки Букли» в `src/components/Pet/PetScreen.jsx`:
- Прогресс-бар привязанности + текст «Корми Буклю, чтобы получить подарок».
- При `bond ≥ BOND_PER_GIFT` и наличии несобранного подарка → светящаяся кнопка
  **«Букля что-то принесла! Забрать»** + индикатор-точка на иконке питомца в
  шапке (`Header`).
- Тап «Забрать» → анимация открытия → следующий по порядку несобранный id
  добавляется в `prefs.petGifts`, `bond → 0`. **Без автоприменения.**
- Сетка коллекции: собранные показаны (иконка/название), несобранные —
  силуэты «?». Тап по собранному → **применить** как косметику (ставит
  `activeBackground`/`activeCellStyle` = id подарка); повторный тап/«Стандартный»
  — снять, как в магазине.
- При исчерпании каталога — строка «Скоро новые подарки».

Реген энергии от сытости остаётся отдельным мгновенным бонусом (как сейчас).

## Анти-эксплойт / крайние случаи

- Bond капится на `BOND_PER_GIFT` (нельзя накопить несколько подарков впрок).
- Низкая сытость → почти нет прогресса; offline голод быстро обнуляется.
- Клиентские часы: подарки чисто косметические, без экономической ценности —
  серверная защита не требуется.
- Каталог исчерпан → bond заморожен, петля корректно «спит» до новых элементов.
- Питомец не вылупился (gate 10 игр) → секция подарков скрыта/недоступна.

## Проверка

1. `npm run build` зелёный.
2. Накопление: пока голод > 0, bond растёт ровно +1/5мин (12/час) — одинаково
   при сытости 5 и 100 (проверить мока `now`); при голоде 0 — не растёт;
   офлайн-окно засчитывает только время, пока голод оставался > 0.
3. Выдача: при `bond≥100` появляется кнопка «Забрать»; тап → подарок в
   коллекции, bond=0, **косметика не применилась сама**.
4. Применение: тап по собранному подарку ставит фон/стиль; «Стандартный» снимает.
5. Персистентность: применённый подарок и собранная коллекция переживают F5
   (web) и перезаход (Яндекс-облако); reconnect не затирает.
5a. Смена устройства / очистка localStorage: собрать подарки на устройстве A →
    на устройстве B (или после очистки localStorage) коллекция подтягивается с
    сервера (union), НЕ обнуляется и НЕ затирает серверный список пустым
    локальным. Проверить обе стороны гонки (поздний `userId`).
6. Исчерпание: собрать все 4 → строка «Скоро новые подарки», bond заморожен.
7. Реген энергии от сытости работает как раньше.

## Вне рамок (YAGNI)

- Редкости/рандом-дроп (пока строго по порядку).
- Подарки-консумаблы (подсказки/энергия) — только косметика.
- Серверная авторитетность bond.
